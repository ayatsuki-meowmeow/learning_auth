### 1. Openapi.yamlのスキーマ定義

- 認証情報のスキーマとしてAuthを作る。
- 認証情報の登録はユーザー登録と同時にするという前提での設計とする。 
- したがって、リクエストはユーザー情報 + 認証情報となる。
- レスポンスには秘匿情報を絶対に含めない。

### 2. DBテーブルのスキーマ定義

- テーブルはユーザー情報と別で認証情報を持つ。
- 認証情報テーブルはユーザーのIDを外部キーで持つようにする。
- ユーザーIDを使って検索してパスワードの正しさを照合する、かつ、ユーザーIDは当然ユニークなので、ユーザーIDにIndexを貼っておくとパフォーマンスが良くなる。
- Drizzle-ormでのリレーション定義は、親と子の双方で定義が必要なので循環参照となってしまう。回避のために、リレーションのみを定義するファイルを別で作るのが良い。
  - https://orm.drizzle.team/docs/relations
  - ただし、Drizzle-ormのversion 1以降はv2の定義をすることで循環参照を回避できる。
    - https://orm.drizzle.team/docs/relations-v2

### 3. ハッシュ化

- ハッシュ化とは任意の値を固定長の文字列に変換すること。ハッシュ化する関数がハッシュ関数
  - ハッシュ値から元の値に戻すことはできない
  - 同じ入力からは常に同じハッシュ値になる
- パスワードをそのままDBに保存すると、DBが流出したときにパスワードがバレてしまう。
  - なので、DBにはハッシュ値を保存する
  - ログイン時は、入力された値をハッシュ化してDBの情報と照合する
- とはいえ、同じ入力からは常に同じハッシュ値になるという問題がある
  - パスワードとその値のハッシュ値の対応表を使えば逆引きでパスワードを知ることが可能になる
    - この対応表が、レインボーテーブル
  - そこで、ソルトを使う
- ソルトとは、ハッシュ化前にパスワードに付加するランダムな文字列
  - パスワードごとにペアとなるランダムな文字列がソルトとなる
  - パスワード + ソルト の文字列をハッシュ化する
  - ソルトがあると、パスワードごとに再計算が必要となり、レインボーテーブルの準備が非現実的になってくる
  - とはいえ、総当たり攻撃を防ぐにはより高度な防衛手段が必要

#### SHA256
  - 入力値の長さに関わらず、必ず256bitのハッシュ値を返すハッシュ関数
  - GPUを使うと1秒で数十億回計算可能なので、短いパスワードなら当てられてしまう
  - 総当たり攻撃を防げないのでパスワードのハッシュ化には不適

#### bcrypt
  - パスワードにソルトを与えてからハッシュ化するハッシュ関数
  - 同じパスワードでも毎回別のソルトが与えられる
  - ソルトがあるため、レインボーテーブル攻撃対策がされている
  - また、わざと計算を重くしている
  - SHA256と比較すれば十分に遅い(=総当たりにかかる時間が長い)が、GPU並列処理等には弱い
  - Railsの`device`gemで採用されている


#### Argon2id
  - パスワードハッシュアルゴリズムのコンペで優勝したハッシュ関数
  - CPUのみでなくメモリへの負荷も高い
  - そのため、GPUの並列処理でもbcryptと比較してさらに時間がかかる
